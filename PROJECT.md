## 🧠 開発で得た教訓まとめ

### 技術的な教訓

#### Phaserの落とし穴（day00004で発見）

**問題**: 
- `scene.tweens.add()` を使いながら、同時に `update()` でオブジェクトの位置を変更すると**クラッシュする**
- 配列操作との競合により、フレームが完全にフリーズ

**解決策**:
- **Canvas APIを使う**: 素のCanvas APIなら予測可能な動作
- tweenを避け、単純な値の変更のみ（`ctx.fillRect()`, `ctx.drawImage()` など）

**結論**: **day00005以降はCanvas APIを推奨**

#### 音ゲーの設計ノウハウ（day00005で学習）

**BPMとノーツ間隔**:
- BPM120 → 4分音符=500ms、8分音符=250ms、16分音符=125ms
- **人間の反応限界**: 16分音符（125ms間隔）は押せない → **8分音符（250ms）が最小単位**

**判定窓の設計**:
- Perfect: ±50ms（厳しめ）
- Good: ±100ms（普通）
- Miss: それ以外
- **判定は早押し優先**: 一度判定したノーツはスキップ

**難易度設計**:
- **イントロは空白**: カウントダウン後、数拍（2〜4拍）は空白にする
- **最初のノーツが早すぎると反応できない**: プレイヤーがリズムに慣れる時間が必要

**フィーバーモードで差別化**:
- 連続正解でフィーバーモード発動
- 背景が変わる、スコア倍率が上がる演出
- ユニーク要素でゲームに個性を出す

### ゲームデザインの教訓

#### 「運ゲー」は致命的（day00004で学習）

**問題**: 
- 完全ランダムで不良品が流れるゲームは、スキルで上達できない
- プレイヤーは「運が悪かった」と感じて離脱

**解決策**:
- **パターンを覚える要素を追加**: 「赤い製品は不良品」など、学習できるルール
- **反応速度を要求**: ランダム性よりも、素早く正確に判断する要素を強化

#### 育成（成長実感）は重要（day00004で学習）

**良い点**:
- ロボット機能（自動で良品を通す）は魅力的
- プレイヤーが「成長している」実感を得られる

**注意点**:
- パワーアップが不要なゲームもある（リズムゲーなど）
- ゲームのコアループに合わせて育成要素を設計

#### スマホテスト必須（day00001〜day00005で継続学習）

**問題**:
- 自動テストでは見つからないバグがある
- タップ判定のずれ、画面サイズの問題、効果音の遅延など

**解決策**:
- **WIP環境でスマホテスト**: GitHub Pagesにpushして実機で確認
- **改善ループ**: 2〜3回テスト → 修正 → 再テスト

### 運用の教訓

#### 命名規則（day00001から継続）

**URL（機械用）**:
- フォーマット: `day00001` ~ `day99999`
- 5桁ゼロパディング
- 理由: URL直接編集でのアクセス対応、27年分（9999日）確保

**表示名（人間用）**:
- フォーマット: `1日目` ~ `99999日目`
- `index.html` で JavaScript が自動変換

#### cronを使った自動改善ループ（day00004で導入）

**手法**:
1. **cron設定**: 毎日決まった時間（例: 21:00）に改善ループを実行
2. **改善指示テンプレート**: 前日のゲームを分析し、3つの改善案を提示
3. **イテレーション開発**: 改善案をAIに伝え、次のバージョンを作成
4. **自動commit**: `improvements.md` に記録してGitにpush

**メリット**:
- 人間が指示を出さなくても、AIが自律的に改善を提案
- 改善履歴が残るので、長期的な学習が可能

**設定例**:
```bash
# crontab -e
0 21 * * * cd /home/ageha/.openclaw/workspace && openclaw session run --session agent:main "最新ゲームを分析して3つの改善案を提示、improvements.mdに追記してpush"
```

### 前提条件

このドキュメントを読む別サーバーのAIは、以下のスキルを参照して作業を行います：

1. **minigame-daily**: 毎日の開発・公開フロー
2. **game-design**: ゲームデザイン・プロトタイプ作成
3. **game-deploy**: GitHub Pagesへのデプロイ
4. **youtube-upload**: YouTube Shorts へのアップロード

### 基本的な1日のフロー

#### 前日 18:00-22:00: 開発期間

1. **アイデア出し**: `improvements.md` を参照
2. **WIP環境でプロトタイプ作成**: Canvas API推奨
3. **テストプレイ（PC + スマホ）**: GitHub Pages WIP環境で実機テスト
4. **改善イテレーション**: 2〜3回繰り返す
5. **最終調整**: WIPに保存

#### 当日 06:00-07:00: 公開準備

1. **最終チェック**: WIP環境で動作確認
2. **デプロイ**: `./deploy.sh` で本番環境にリリース
3. **動画録画**: スマホで縦長・60秒以内
4. **YouTube Shorts アップロード**: `youtube_uploader.py`
5. **Twitter下書き生成**: `twitter_draft.py`
6. **Twitter投稿**: **人間が手動で投稿**（Bot回避）

#### 当日 21:00: 振り返り

1. **improvements.md に記録**: 良かった点、改善点、次回の候補
2. **cronによる自動改善**: AIが自律的に改善案を提示（オプション）

## 📝 注意事項（別サーバーのAI向け）

### 絶対に守ること

1. **Twitter投稿は人間が手動で行う**: AI が自動投稿すると Bot 扱いされて収益化不可
2. **ファイルは必ず書き込む**: メンタルノート不可、すべて `improvements.md` や Git に記録
3. **スマホテスト必須**: WIP環境で実機テスト、自動テストだけでは不十分
4. **Canvas API 推奨**: Phaser.js は tween + 配列操作で落とし穴が多い

### 推奨事項

1. **cron で自動改善ループ**: 毎日21:00に前日のゲームを分析し、改善案を提示
2. **ユニーク要素で差別化**: フィーバーモード、育成要素など、他にはない要素を追加
3. **スキルで上達できる要素**: ランダム性よりも、反応速度や判断力を要求
4. **イテレーション開発**: 2〜3回の改善ループで完成度を上げる
