# ゲームデザイン原則

## 📊 ゲームデザインの基本要素

### 1. コアループ（Core Loop）
**定義**: プレイヤーが繰り返し行う基本的な行動の循環

**良いコアループの条件**:
- 短い（数秒〜数十秒）
- 明確（何をすれば良いか分かる）
- 報酬がある（達成感、進行感）
- 繰り返したくなる

**例: Tetris**
```
1. ブロックが落ちてくる
2. 回転・移動して配置
3. 列が揃う → 消える（報酬）
4. 次のブロック → 繰り返し
```

---

### 2. メカニクス（Mechanics）
**定義**: プレイヤーができる具体的な操作

**良いメカニクスの条件**:
- 直感的（説明不要で理解できる）
- レスポンシブ（操作→反応が速い）
- 習熟性がある（上手くなる実感）

**例**:
- タップでジャンプ
- スワイプで移動
- 長押しで溜める

---

### 3. 難易度曲線（Difficulty Curve）
**定義**: 時間経過に伴う難易度の変化

**良い難易度曲線**:
```
難易度
 ↑
 |    /
 |   /
 |  /
 | /
 |/________→ 時間
```

**重要ポイント**:
- 最初は簡単（学習期間）
- 徐々に難しく（挑戦）
- プレイヤーの成長と並行

**悪い例**:
- 最初から難しすぎる → 挫折
- ずっと簡単 → 飽きる
- ランダムに難易度が変わる → 不公平感

---

### 4. フィードバック（Feedback）
**定義**: プレイヤーの行動に対する反応

**種類**:
1. **即時フィードバック**: 操作直後（効果音、エフェクト）
2. **進行フィードバック**: スコア、距離、時間
3. **達成フィードバック**: ランク、記録更新

**例**:
- ✅ タップ → 音が鳴る → 「反応した」と分かる
- ✅ 正解 → 光る → 「良いことをした」と分かる
- ❌ 何も反応がない → 「押せてる？」と不安

---

### 5. プレイヤーファンタジー（Player Fantasy）
**定義**: プレイヤーが体験したい気分・役割

**例**:
- 「速く走りたい」→ ランナーゲーム
- 「正確に狙いたい」→ シューティング
- 「賢く選びたい」→ パズル

**重要**: ファンタジーとメカニクスが一致すること
- ✅ 走る → 前進する背景
- ❌ 走る → 静止画面

---

## 🎯 良いゲームの条件（ミニゲーム向け）

### 必須要素
1. **10秒以内に理解できる**
   - 説明不要が理想
   - 最長でも1画面の説明

2. **スキルで上達できる**
   - 運要素は最小限
   - 練習すれば上手くなる実感

3. **1プレイが短い（30秒〜2分）**
   - 気軽に遊べる
   - もう1回！と思える

4. **明確な目標**
   - 何を目指すのか分かる
   - スコア、距離、時間など

5. **即座のフィードバック**
   - 操作→反応が速い
   - 良い/悪いが分かる

### あると良い要素
1. **進行感**
   - 「前に進んでる」感覚
   - 背景スクロール、数値増加

2. **リスク/リターン**
   - ギリギリを狙うと高得点
   - 安全策は低得点

3. **記録更新の喜び**
   - ハイスコア保存
   - ランク表示

---

## ❌ 避けるべき要素

### 1. 運ゲー
- ランダム性が強すぎる
- スキルで勝てない
- 何度やっても同じ

**例**:
- ❌ ランダムな箱を選ぶだけ
- ✅ 情報を見て判断する

### 2. 不公平感
- 避けられない障害
- 反応不可能な速度
- 見えない攻撃

### 3. ストレス
- 操作が効かない
- 反応が遅い
- 何をすれば良いか分からない

---

## 🔧 開発フロー

### 1. コンセプト決定
```
Q1. プレイヤーは何を体験する？（ファンタジー）
Q2. 基本操作は何？（メカニクス）
Q3. 繰り返す行動は？（コアループ）
Q4. 目標は何？（ゴール）
```

### 2. プロトタイプ
- **最小限の実装**
- コアループだけ作る
- 見た目は後回し

### 3. 自己テスト
- **10回プレイ**
- 飽きないか？
- もう1回やりたいか？
- バグはないか？

### 4. 改善
- 難易度調整
- フィードバック追加
- エフェクト強化

### 5. リリース
- 最終チェック
- スマホでテスト
- デプロイ

---

## 📝 チェックリスト

### デプロイ前
- [ ] 10回以上自分でプレイした
- [ ] バグがない（クラッシュ、無限ループ等）
- [ ] 操作が直感的
- [ ] スマホで動作確認
- [ ] ハイスコアが保存される
- [ ] 効果音がある
- [ ] 説明が分かりやすい

### ゲーム性
- [ ] コアループが楽しい
- [ ] スキルで上達できる
- [ ] 難易度が適切
- [ ] フィードバックが即座
- [ ] 目標が明確

## 🎮 成功例の分析

### Flappy Bird
- **コアループ**: タップ→飛ぶ→避ける→通過→繰り返し
- **メカニクス**: タップのみ
- **難易度**: 一定（パイプ間隔固定）
- **フィードバック**: 音、スコア
- **ファンタジー**: 「ギリギリを攻める緊張感」

**成功理由**:
- シンプルすぎる操作
- 明確な目標（スコア）
- 即死の緊張感
- もう1回！と思える短さ

### Tetris
- **コアループ**: ブロック→配置→消す→繰り返し
- **メカニクス**: 回転、移動
- **難易度**: 速度上昇
- **フィードバック**: 列が消える爽快感
- **ファンタジー**: 「完璧に整理する快感」

**成功理由**:
- 消す爽快感
- スキルで上達できる
- 難易度が自然に上がる
- 終わりがない（エンドレス）

## 💡 今後の方針

### 開発時
1. まず理論を確認
2. コンセプトを明確に
3. プロトタイプを作る
4. **自分で10回テスト**
5. バグ修正
6. マリモに提供

### テスト時の観点
- 操作ミスはないか？
- 連打バグはないか？
- ゲームオーバー後のリセットは正常か？
- スコアは正しく表示されるか？
- ハイスコアは保存されるか？

---

**重要**: 面白くないと思ったら、素直に作り直す。無理に改良しない。
